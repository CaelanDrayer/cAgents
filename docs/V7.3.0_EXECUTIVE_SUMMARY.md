## Architecture Decision

### Option A: New "Play" Super-Domain ✅ (RECOMMENDED)

**Structure**:
```
Super-Domains (6):
├── Make: Engineering + Product
├── Grow: Marketing + Sales
├── Operate: Finance + Operations
├── People: HR + Culture
├── Serve: Customer + Legal + Compliance
└── Play: Game Development + Interactive Entertainment (NEW)
```

**Play Domain Composition**:
- **Controllers** (3): game-designer, game-producer, technical-director
- **Execution Agents** (25):
  - Design (7): mechanics, level, systems, economy, progression, UX, balancer
  - Programming (7): gameplay, graphics, AI, physics, networking, tools, engine
  - Content (6): technical artist, 3D, 2D, animator, audio, VFX
  - Production (3): project manager, QA tester, playtest coordinator
  - Narrative (2): quest designer, interaction designer

**Why Play Domain?**:
- ✅ Clean separation from general creation (Make)
- ✅ Game dev is distinct professional field
- ✅ Room for expansion (VR/AR, esports, mobile)
- ✅ Matches industry structure
- ✅ Clear routing logic

**Alternative (Rejected)**: Expand Make domain
- ❌ Make becomes too large (105 agents)
- ❌ Game dev workflows might get lost
- ❌ Less clarity for game-specific work

## Technical Architecture

### V7.0 Controller-Centric Pattern

**Workflow Example**: "Create a roguelike deck-building game"

```
1. Routing → Play domain detected (keywords: "roguelike", "game")
2. Planning → Tier 3 (complex), Controllers: game-designer + technical-director
3. Coordinating:
   - game-designer asks: "What is core gameplay loop?" → mechanics-designer
   - game-designer asks: "What card types?" → systems-designer
   - technical-director asks: "What engine?" → game-engine-specialist
   - Controllers synthesize answers into implementation plan
4. Executing:
   - gameplay-programmer implements card system
   - level-designer creates procedural generation
   - 2d-artist creates card art
5. Validating:
   - game-qa-tester playtests
   - Validation passes → workflow complete
```

**Key Innovation**: Controllers use question-based delegation to coordinate specialists, not direct task assignment.

## Key Workflows Supported

### Workflow 1: Game Concept to Prototype (Tier 3)
**Example**: "/trigger Create a roguelike deck-building game prototype"
- Concept → Design → Prototyping → Testing
- 5-10 hour workflow
- Deliverable: Playable prototype with core mechanics

### Workflow 2: Mechanics Design and Implementation (Tier 2)
**Example**: "/trigger Design and implement a combo system for fighting game"
- Design → Documentation → Implementation → Testing
- 2-4 hour workflow
- Deliverable: Working mechanic with tests

### Workflow 3: Level/Content Creation (Tier 2)
**Example**: "/trigger Create 5 dungeon levels for RPG"
- Design → Blockout → Art pass → Scripting → Testing
- 3-6 hour workflow
- Deliverable: 5 completed levels

### Workflow 4: Game Engine Integration (Tier 3)
**Example**: "/trigger Set up Unity project for 2D platformer"
- Setup → Core systems → Gameplay systems → Integration
- 4-8 hour workflow
- Deliverable: Configured Unity project with systems

### Workflow 5: Game Balancing (Tier 2)
**Example**: "/trigger Balance weapon damage values based on playtest data"
- Data collection → Analysis → Adjustment → Testing
- 1-2 hour workflow
- Deliverable: Balanced gameplay with data-driven changes

### Workflow 6: Narrative Game Development (Tier 3)
**Example**: "/trigger Create main story questline for fantasy RPG"
- Story design → World design → Quest design → Dialogue → Integration
- 6-12 hour workflow
- Deliverable: Complete questline with branching narrative
